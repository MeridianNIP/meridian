Global Settings
Super-admin-only · portal-wide policy root · changes apply to all users / groups immediately and are recorded to the tamper-evident audit log.
Why this page matters: Commercial / regulated deployments (SOC 2, HIPAA, ISO 27001, PCI-DSS) require a single, documented source of portal-wide security policy. Settings here cap what admins and users can do -- sub-roles can tighten (opt into a shorter idle timeout, stronger MFA) but cannot loosen beyond what a super-admin sets here.
Preview / design surface — most controls are not yet editable. The underlying policy schemas (roles, role_permissions, audit_retention, compliance_baselines) haven't shipped yet. In the meantime, edit live policies from their current home:
  • Session idle timeout, password policy, MFA requirement, lockout — Admin → Branding (these live on the branding row for now)
  • Roles assigned to users — Admin → Users
  • Audit retention / compliance baselines — no edit surface yet; pending schema work.
The Save buttons below are intentionally disabled with a per-control tooltip explaining what's blocked.
Sessions & idle timeout
Global default + maximum idle timeout (in minutes) for every user. A user may set their own value lower via Settings → Session, but never higher than the maximum below. Setting the default to 0 disables idle logout entirely (not recommended for regulated environments).

Scope Manager is a separate, unrelated system — it controls which targets (domains / IPs / CIDRs) sandboxed tools may query, not what actions a user can perform. For AD auth, map each AD group to exactly one built-in role under Admin → Integrations → Directory → [role map]; the user inherits whichever role is highest-privilege among their AD groups on every login.
Role Intent Users (local)
Custom role definitions + editable permission matrix are a future schema change (will go into roles + role_permissions tables). The built-in set covers the common tiers — ask for a custom role to be added in a code release if your org needs an extra split.
Password policy
Minimum strength requirements enforced on every password change. The portal uses argon2id for hashing and always rejects the NIST-blocklist common passwords on top of whatever is configured here.
MFA policy
TOTP (authenticator app) is always allowed. WebAuthn/FIDO2 is on the roadmap (server flow not wired yet).

Audit retention
Audit events are HMAC-chained (tamper-evident). Retention applies to the audit_events table; chained hashes survive the archive so integrity can still be verified on archived batches.

Log shipping destinations
loading…
Ship the tamper-evident audit log to an external collector. Up to three destinations. Supports syslog (RFC 5424, TCP/UDP/TLS), Splunk HEC, Elastic _bulk, and CEF-over-syslog (ArcSight/QRadar). Auth secrets live in the vault. A beat task flushes new events every 30s (per-destination cursor so a downed collector doesn't drop events — they resume when it comes back).
